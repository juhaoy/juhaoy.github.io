AOSP 基本概念介绍

Posted on 2025-03-16 | In Android 开发

lunch 编译类型

在 Android 开源项目（AOSP）的编译环境中，lunch 命令会列出一系列的编译组合选项，如下所示：

每个选项都代表了不同的目标设备、编译类型和平台架构的组合。以下是对这些选项中常见部分含义的详细解释：

目标设备类型

  • 通用设备（aosp_*）
    • aosp 代表 Android Open Source Project，即通用的AOSP编译配置，适用于多种类型的设备。例如 aosp_arm-eng 表示基于 AOSP 为 ARM 架构设备编译的通用配置。
  • 汽车设备（aosp_car_*）
    • 以 aosp_car_ 开头的选项是专门为汽车平台设计的 Android 系统编译配置。例如 aosp_car_arm64-userdebug 是为64位 ARM 架构的汽车设备编译的配置。
  • 模拟器设备（mini_emulator_* 和 m_e_*）
    • mini_emulator_* 和 m_e_* 是用于模拟器的编译配置。例如 mini_emulator_x86_64-userdebug 是为64位 x86 架构的模拟器编译的配置。
  • UML（uml-*）
    • uml 代表 User Mode Linux，是一种在用户空间运行的 Linux 内核，这里表示为 UML 环境编译的 Android 系统配置。
  • 特定设备（aosp_marlin、aosp_sailfish等）
    • 像 aosp_marlin、aosp_sailfish 等选项是针对特定硬件设备的编译配置。例如 aosp_marlin 是为 Google Pixel 系列设备编译的配置。
  • CF（aosp_cf_* 和 cf_*）
    • cf 可能代表某种特定的设备配置或功能集，通常用于特定的测试或开发场景。例如 aosp_cf_x86_auto-userdebug 是为 x86 架构的汽车设备特定功能集编译的配置。

平台架构

  • arm：32位 ARM 架构，常用于大多数移动设备。
  • arm64：64位 ARM 架构，现代高端移动设备普遍采用。
  • mips：32位 MIPS 架构，曾经用于一些嵌入式设备和智能手机。
  • mips64：64位 MIPS 架构。
  • x86：32位 x86 架构，常用于模拟器和一些基于 x86 的设备。
  • x86_64：64位 x86 架构，同样适用于模拟器和基于 x86 的设备。

编译类型

  • eng：开发版本，包含了大量的调试信息和开发工具，方便开发者进行调试和开发工作。
  • userdebug：用户调试版本，既包含了一些调试功能，又接近最终用户使用的版本。通常用于内部测试和调试。
  • user：最终用户版本，去除了大部分调试信息，以提高系统性能和安全性。

示例解释

  • aosp_arm-eng：基于 AOSP 为32位 ARM 架构设备编译的开发版本。
  • aosp_car_x86_64-userdebug：基于 AOSP 为64位 x86 架构的汽车设备编译的用户调试版本。
  • mini_emulator_x86-userdebug：为32位 x86 架构的模拟器编译的用户调试版本。
